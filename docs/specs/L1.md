# L1 - High-Level Requirements

These are the high-level capabilities that **The Upper Room** (a platform for city leads of the FaithTech faith-tech organization) must provide. Every L2 requirement traces back to one of these.

---

## L1-001: User Authentication via PKCE Authorization Code Flow
The system must authenticate users through the OAuth 2.0 Authorization Code flow with Proof Key for Code Exchange (PKCE), with no implicit flow and no use of client secrets in the browser.

## L1-002: User Account and Profile Management
The system must allow users to register, verify their email, sign in, sign out, recover their password, change their password, edit their profile, upload an avatar, and delete their account.

## L1-003: Role-Based Access Control (RBAC)
The system must enforce role-based access control end to end (database, backend authorization policies, frontend route guards and component-level visibility), with at minimum the roles `SystemAdmin`, `CityLead`, `Member`, and `Guest`.

## L1-004: Contact Management
The system must allow authorized users to create, read, update, archive, and delete contact records, each with basic info, address, multiple phone numbers, multiple emails, tags, and free-form notes.

## L1-005: Partner Organization Management
The system must allow authorized users to create, read, update, archive, and delete partner organization records, each with basic info, address, multiple phone numbers, multiple emails, links to contact records, tags, and free-form notes.

## L1-006: Tagging System
The system must provide a unified tagging mechanism that can be applied across contacts, partners, ideas, and events, with tag CRUD, color assignment, and filter-by-tag everywhere tags appear.

## L1-007: Notes
The system must allow authorized users to attach time-stamped, authored, free-form notes (Markdown) to contacts, partners, ideas, and events, with edit history.

## L1-008: Configurable Kanban Boards
The system must provide kanban boards whose boards, columns, swimlanes, and card schemas are stored in the database and configurable at runtime, supporting drag-and-drop of cards across columns using Angular CDK drag-and-drop.

## L1-009: Hackathon Idea Management
The system must allow authorized users to capture, refine, vote on, and track hackathon ideas, optionally associated with one or more partner organizations, and surface them on a dedicated kanban board.

## L1-010: Event and Meetup Management
The system must allow authorized users to create, read, update, cancel, and delete events and meetups, with status (Potential, Scheduled, In Progress, Past, Cancelled), capacity, optional location, attendee tracking, and RSVP functionality.

## L1-011: Location Management
The system must allow authorized users to create, read, update, archive, and delete reusable physical locations (name, address, capacity, accessibility info, photos) that can be referenced from events.

## L1-012: Calendar View
The system must provide a calendar view (month, week, day, agenda) of events, filterable by status, tag, partner, and location, accessible to all authenticated users.

## L1-013: City-Lead Workspace Scoping
The system must scope data by City so that each City Lead sees and manages only their city's data by default, while SystemAdmins can switch cities or view across cities.

## L1-014: Mobile-First Responsive Design
The system must deliver a responsive web application that works at the XS (<576px), SM (>=576px), MD (>=768px), LG (>=992px), and XL (>=1200px) breakpoints, designed mobile-first and progressively enhanced for larger screens.

## L1-015: Material Design 3 Visual Language
The system must adhere to the Material Design 3 (M3) specification (https://m3.material.io/) for color, typography, elevation, motion, shape, and component behavior, using `@angular/material` v17+ with M3 theming.

## L1-016: Notifications and Feedback
The system must provide consistent in-app notifications (snackbar toasts, banners, dialogs, badges) for all asynchronous outcomes, with prescribed copy, duration, severity, and dismissal behavior.

## L1-017: Error Handling and Messaging
The system must surface every error with a user-readable message, an actionable recovery hint, a unique correlation ID, and a defined severity, while logging full diagnostic details server-side.

## L1-018: Accessibility (WCAG 2.2 AA)
The system must meet WCAG 2.2 Level AA across all pages, components, and flows -- including keyboard navigation, screen reader semantics, color contrast, focus visibility, and reduced-motion support.

## L1-019: Performance
The system must meet Core Web Vitals targets (LCP <= 2.5s, INP <= 200ms, CLS <= 0.1) on a mid-range mobile device on a 4G connection, and serve all API requests within 300ms p95 under expected load.

## L1-020: Security and Data Protection
The system must protect data at rest and in transit, validate every input on both client and server, defend against the OWASP Top 10, enforce CSRF and CORS correctly, and never log secrets or PII.

## L1-021: Observability and Auditing
The system must emit structured logs, metrics, and traces, and write an immutable audit trail for every create, update, delete, login, and authorization decision.

## L1-022: Backend Architecture (.NET Clean Architecture)
The backend must be a .NET solution following Clean Architecture with the project layout `backend/TheUpperRoom.sln` (old-style format), `backend/src` for production projects (TheUpperRoom.Api, TheUpperRoom.Application, TheUpperRoom.Domain, TheUpperRoom.Infrastructure), `backend/tests` for tests, the free version of MediatR for CQRS, and Microsoft SQL Server as the database. All backend code must adhere to S.O.L.I.D. design principles (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion) and must use the Microsoft.Extensions.* libraries as the standard primitives for cross-cutting concerns: `Microsoft.Extensions.Logging` (logging), `Microsoft.Extensions.Configuration` (configuration), `Microsoft.Extensions.DependencyInjection` (DI/IoC), `Microsoft.Extensions.Options` (strongly-typed options with validation), `Microsoft.Extensions.Hosting` (generic host/background services), and `Microsoft.Extensions.Http` (typed `HttpClient` factory). No bespoke logger, config, DI container, or options abstraction may be introduced.

## L1-023: Frontend Architecture (Angular Workspace)
The frontend must be an Angular workspace partitioned into an `api` library (models + services with contract files), a `components` library (reusable UI primitives, no api dependency), a `domain` library (UI components that depend on `api`), and a `the-upper-room` application that depends on all three; every component must be split into separate `.html`, `.scss`, and `.ts` files (no inline templates or styles), and all CSS class names must follow BEM.

## L1-024: Testing
The system must have unit tests on backend (xUnit) and frontend (Jest or Karma), and Playwright end-to-end tests using the Page Object Model for every important user journey.

## L1-025: Internationalization Readiness
The system must externalize all user-facing copy through i18n keys so that locales can be added without code changes; English (en-CA) is the default and only required locale at launch.

## L1-026: Application Shell and Navigation
The system must provide a consistent application shell with a top app bar, a navigation drawer (modal on XS/SM, persistent on MD+), a breadcrumb trail, a global search affordance, a user menu, and a content area, with prescribed transitions between routes.

## L1-027: Search and Filtering
The system must provide list-page filtering, sorting, and searching, plus a global search reachable from the top app bar that searches across contacts, partners, events, ideas, and locations.

## L1-028: Data Validation
The system must enforce identical validation rules on the client (Angular reactive forms) and server (FluentValidation) for every form field, with prescribed messages and ARIA-compliant error display.

## L1-029: Offline and Network Resilience
The system must detect network loss, surface an offline banner, queue idempotent mutations where safe, and automatically retry transient failures with exponential backoff.

## L1-030: Empty, Loading, Error, and Skeleton States
Every list, detail, dashboard, and form view must define and render distinct Empty, Loading (skeleton), Error, and Success states with prescribed copy and visuals.
